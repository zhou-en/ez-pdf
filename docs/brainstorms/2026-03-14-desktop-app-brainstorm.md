# Desktop App — macOS & Linux (Phase 20)

## What We're Building

A native desktop application for macOS and Linux that wraps the five core ezpdf
operations (merge, split, remove, rotate, reorder) in a polished drag-and-drop UI.
The app lives in the existing `ezpdf-app` Cargo workspace member.

**Stack chosen:** Tauri v2 + Svelte 5 + Vite
**Target platforms:** macOS (arm64 + x86_64), Linux (x86_64)
**Scope:** Core 5 operations only — merge, split, remove, rotate, reorder

## Why Tauri v2 + Svelte 5

Alternatives considered and rejected:

| Option | Rejected because |
|--------|-----------------|
| egui (pure Rust, immediate-mode) | Looks non-native, limited layout primitives |
| iced (pure Rust, Elm-like) | Less mature, harder to achieve polished look |
| Tauri + plain HTML/vanilla JS | Harder to maintain, no component model |
| Electron | 200 MB+ bundle, defeats "fast + lean" brand |

**Tauri v2 wins because:**
- Ships a ~10–15 MB bundle (vs Electron's 200 MB)
- Uses the OS webview (WKWebView on macOS, WebKitGTK on Linux)
- Svelte 5 compiles to minimal JS — no framework runtime overhead
- Tauri commands (IPC) let us call ezpdf-core directly from Rust
- Genuine native OS integration: file dialogs, drag-and-drop, menu bar

## User Experience

**Interaction pattern:** Drag-and-drop with sidebar operation picker

```
┌──────────────────────────────────────────────┐
│  ezpdf                              [─][□][×] │
├──────────┬───────────────────────────────────┤
│          │                                   │
│ Merge    │    Drop PDFs here, or click       │
│ Split    │         to browse                 │
│ Remove   │                                   │
│ Rotate   │   [file1.pdf ×]  [file2.pdf ×]   │
│ Reorder  │                                   │
│          ├───────────────────────────────────┤
│          │  Output: [same folder as input ▾] │
│          │  [        Run Merge         ]     │
│          │                                   │
│          │  ✓ Merged → report.pdf (1.2 MB)   │
└──────────┴───────────────────────────────────┘
```

- Left sidebar: select operation (highlights active)
- Main panel: drop zone + file list (files reorderable for Reorder op)
- Options panel: operation-specific options (pages for Split/Remove/Rotate)
- Output: "same folder as first input file" default + "Save As…" override
- Status bar: last action result (success/error) with file size

## Architecture

### Workspace layout

```
ezpdf-app/
├── Cargo.toml          ← add tauri dependency
├── src/
│   ├── lib.rs          ← Tauri commands (the testable layer)
│   └── main.rs         ← app entry point (calls tauri::Builder)
└── frontend/           ← Svelte 5 + Vite app
    ├── package.json
    ├── vite.config.ts
    ├── src/
    │   ├── App.svelte
    │   ├── components/
    │   │   ├── Sidebar.svelte
    │   │   ├── DropZone.svelte
    │   │   ├── FileList.svelte
    │   │   └── OptionsPanel.svelte
    │   └── lib/
    │       └── tauri.ts    ← typed wrappers around invoke()
    └── tests/              ← Vitest component tests
        └── *.test.ts
```

### Tauri command → ezpdf-core mapping

```rust
// ezpdf-app/src/lib.rs

#[tauri::command]
pub fn cmd_merge(inputs: Vec<String>, output: String) -> Result<String, String>

#[tauri::command]
pub fn cmd_split(input: String, range: String, output: String) -> Result<String, String>

#[tauri::command]
pub fn cmd_remove(input: String, pages: String, output: String) -> Result<String, String>

#[tauri::command]
pub fn cmd_rotate(input: String, degrees: i32, pages: Option<String>, output: String) -> Result<String, String>

#[tauri::command]
pub fn cmd_reorder(input: String, order: String, output: String) -> Result<String, String>
```

All commands:
- Accept String paths (Tauri IPC serialises to JSON)
- Return `Result<String, String>` — success message or error string
- Delegate to `ezpdf_core::*` functions (same functions the CLI uses)
- Are unit-testable with `#[cfg(test)]` blocks in `lib.rs`

### Tauri v2 capabilities

File system access requires explicit capability grants in Tauri v2:

```json
// src-tauri/capabilities/default.json (auto-generated by tauri init)
{
  "permissions": [
    "core:default",
    "dialog:allow-open",
    "dialog:allow-save",
    "fs:allow-read-files",
    "fs:allow-write-files"
  ]
}
```

## TDD Strategy for Autonomous Implementation

### Backend (Rust Tauri commands)

Standard [RED]/[GREEN]/[REFACTOR] with `cargo test`:

```rust
#[cfg(test)]
mod tests {
    #[test]
    fn cmd_merge_combines_pages() { … }
    #[test]
    fn cmd_merge_missing_input_returns_err() { … }
}
```

`cargo test -p ezpdf-app` runs these in isolation (no Tauri runtime needed).

### Frontend (Svelte 5)

Vitest + @testing-library/svelte for component tests:

```typescript
// frontend/tests/DropZone.test.ts
import { render, fireEvent } from '@testing-library/svelte'
import DropZone from '../src/components/DropZone.svelte'

test('emits files-added event on drop', async () => { … })
```

`pnpm test` (inside `ezpdf-app/frontend/`) runs Vitest.

### Ralph loop adaptation

The ralph PROMPT.md `[REVIEW]` step must run:
```bash
cargo test --workspace
cd ezpdf-app/frontend && pnpm test
```

This is captured in the phase task plan as a modified review step.

## Ralph Implementation Strategy

Because Phase 20 is the most complex phase (two toolchains, no prior GUI code),
the tasks are structured to build bottom-up:

1. **[SETUP]** Scaffold: `pnpm create tauri-app` + configure workspace
2. **[RED]** Tauri command tests (all 5 commands, Rust only)
3. **[GREEN]** Implement Tauri commands calling ezpdf-core
4. **[SETUP]** Svelte 5 component structure (App, Sidebar, DropZone stubs)
5. **[RED]** Vitest component tests for DropZone, FileList, OptionsPanel
6. **[GREEN]** Implement Svelte components to pass component tests
7. **[REFACTOR]** Clean up types, error handling, UI polish
8. **[REVIEW]** Both test suites pass; manual smoke test; `tauri build` succeeds

**Key constraint for ralph:** At [REVIEW], ralph must run `tauri build` and verify
it exits 0. This is the proof that the app is actually shippable.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| GUI scope | Core 5 ops only | Keeps Phase 20 bounded; remaining ops (watermark, bookmarks, etc.) are v3 |
| UI pattern | Drag-and-drop + sidebar | Most natural for file operations; what macOS users expect |
| Frontend framework | Svelte 5 (no build framework like SvelteKit) | Static SPA fits Tauri; no SSR needed |
| Frontend tests | Vitest + @testing-library/svelte | Faster than Playwright for component tests; no browser startup overhead |
| Output location | Same folder as first input (default) + Save As override | Zero-config default, power-user escape hatch |
| IPC serialization | Strings for file paths | Avoids Tauri PathBuf serialization complexity |
| Error handling | Return `Result<String, String>` from commands | Serialises cleanly to JS; UI shows error string directly |

## Open Questions

1. **Tauri v2 project init**: Should ralph use `pnpm create tauri-app` (interactive)
   or `cargo tauri init` + manual Vite setup? The latter is more scriptable.
   → **Resolved:** Use `cargo tauri init --ci` with manual `package.json` creation to
   avoid interactive prompts.

2. **Reorder UI**: The reorder operation requires specifying a custom page order
   (e.g., "3,1,2"). Should this be a text input ("3,1,2") or a drag-sortable list?
   → Text input for Phase 20; drag-sortable is a polish item for a later phase.

3. **Split range vs split each**: Split has two modes. Should the app expose both?
   → **Resolved:** Yes, via a radio/toggle: "Extract range" / "Burst all pages".

4. **File output naming**: For split_each, multiple files are created. Output must be
   a directory. For all other ops, output is a single file. The UI must handle both.
   → **Resolved:** Default to same-folder auto-naming (e.g. `input-merged.pdf`) + "Save As…" override button. For split_each (burst), output is a folder; use folder picker.

5. **Tauri build in CI**: The existing release.yml builds CLI binaries. Should Phase 20
   add a separate `tauri-release.yml` workflow, or extend the existing one?
   → Separate workflow (`tauri-release.yml`) to avoid coupling. Out of scope for Phase 20;
   distribution is a future phase.

## Resolved Questions

- GUI framework: Tauri v2 + Svelte 5 (see above)
- Scope: Core 5 operations
- Backend architecture: Tauri commands call ezpdf-core directly
- UI pattern: Drag-and-drop + sidebar
